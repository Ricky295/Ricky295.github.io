How to use it

One GET endpoint, three ways to ask it something: a digit range, a single digit, or a search for a sequence inside π.

1

Decide what you're asking for

A slice of digits (startdigit / enddigit), one digit at a position (getdigit), or where a sequence first appears (pisearch). Pick exactly one — the API checks them in that order.

2

Send the request

Every call recomputes π from scratch up to whatever it can reach in the time budget, so there's no setup step and no cached state to worry about.

3

Handle the possibility of a partial answer

If your request asks for digits beyond what the budget allowed, you get back however much was actually computed instead of a hard failure. Always check for an error field.

1.

Endpoints

All three share the same base URL and differ only by which parameters you send.

Digit range

?startdigit=0&enddigit=20

Returns the digits of π from index startdigit up to (not including) enddigit. Index 0 is the first digit after the decimal point — the leading "3" is stripped before indexing starts.

startdigit — integer, required.
enddigit — integer, required.
Response
{ "result": "14159265358979323846" }

Single digit

?getdigit=6

Returns the one digit sitting at that index.

getdigit — integer, required.
Response
{ "result": "6" }

Search

?pisearch=14159

Returns the index of the first occurrence of that digit sequence within whatever was computed. Returns -1 if it isn't found in the reachable digits — which doesn't mean it never appears, only that it doesn't appear within the time budget's reach.

pisearch — string of digits, required.
Response
{ "result": 0 }
2.

The time budget

This is the one thing about this API that isn't obvious from the parameter names, so it gets its own section.

compute window25,000 ms

Every request runs the spigot algorithm from scratch for up to 25 seconds, checking the clock every 100 digits. When time runs out, whatever's been computed so far is what you get — there's no precomputed cache sitting behind this.

SituationWhat comes back
Range or digit fits inside 25sNormal result field
Range partially exceeds the budgeterror + partialResult with whatever was reachable
Single digit exceeds the budgeterror naming how many digits were actually reached
Search sequence not found in rangeresult: -1 with a note on how far the search went

Practical upshot: don't ask for digit 200,000 and expect an instant answer. Small ranges near the start of π return quickly; anything ambitious should be requested with the understanding that it may come back partial.

Response shape

Success

Partial (range)

Not found (search)
{ "result": -1, "note": "Not found within the first 84213 digits calculated." }

Indexing

Zero-based, and counted from the first digit after the decimal point — π itself (the leading 3) is never included in any result.
